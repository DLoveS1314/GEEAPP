# Plan: PostgreSQL HexRemap 架构优化

## 结论
本计划将当前“前端内存 GeoJSON 图层”改为“PostgreSQL 持久化六角格数据集”。建议使用一个固定物理数据库 `hexremap`，前端让用户填写的是逻辑数据集名，而不是每次导入都动态创建 PostgreSQL database。

原因：动态创建物理数据库需要高权限、难管理连接池、难迁移、难做跨数据集查询，也会增加误删风险。固定 `hexremap` 数据库 + `hex_datasets` 表记录用户指定名称，是更稳的架构。

## 需要确认的设计决策
1. 物理数据库名采用 PostgreSQL 推荐小写：`hexremap`。UI 可以显示 `hexRemap`。
2. 前端“数据库名字”输入框实际保存为 `dataset_name` / `display_name`，不是 `CREATE DATABASE <用户输入>`。
3. 启用 PostGIS。若不启用 PostGIS，只能用 JSONB 存 geometry，后续按视图范围查询和 zoom 性能会差。
4. SQL 字段使用 `snake_case`：`edge_1` 到 `edge_6`。前端或导出 GeoJSON 时可以映射为 `edge-1` 到 `edge-6`。
5. 用户写的 `landconver` 按现有代码和常用命名修正为 `landcover`。
6. `edge_1` 代表“最北方向起始边”，后续 `edge_2..edge_6` 按逆时针顺序存储。

## 目标行为
1. 用户选择或上传 GeoJSON 后，前端要求输入数据集名。
2. 后端读取 GeoJSON，解析每个 feature 的 `properties.level`、`properties.col`、`properties.row`、`properties.center_lon`、`properties.center_lat`。
3. 后端把六角格写入 PostgreSQL，并返回数据集 ID、名称、feature 数量、bounds。
4. 前端图层列表显示数据库中的六角格数据集。
5. 地图展示从数据库读取的 GeoJSON，而不是只依赖前端内存。
6. DEM 和 landcover 采样结果回写数据库的预留字段。
7. 导出 GeoJSON 时从数据库重建 FeatureCollection。
8. 删除图层时删除数据库中的逻辑数据集和对应格网记录，或按确认策略改为软删除。

## 数据库 Schema

### 数据库与扩展
```sql
CREATE DATABASE hexremap;
CREATE EXTENSION IF NOT EXISTS postgis;
```

实际实施时不在应用中自动创建物理数据库，由用户或本地脚本准备 `hexremap` 数据库。应用只负责在连接到的数据库内创建表和索引。

### 表：`hex_datasets`
用于保存每次导入的逻辑数据集。

```sql
CREATE TABLE hex_datasets (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  source_file text,
  feature_count integer NOT NULL DEFAULT 0,
  bounds geometry(Polygon, 4326),
  visible boolean NOT NULL DEFAULT true,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  deleted_at timestamptz
);
```

### 表：`hex_cells`
用于保存每个六角格。

```sql
CREATE TABLE hex_cells (
  id bigserial PRIMARY KEY,
  dataset_id uuid NOT NULL REFERENCES hex_datasets(id) ON DELETE CASCADE,
  level integer,
  col integer,
  row integer,
  center_lon double precision,
  center_lat double precision,
  dem double precision,
  landcover integer,
  edge_1 jsonb,
  edge_2 jsonb,
  edge_3 jsonb,
  edge_4 jsonb,
  edge_5 jsonb,
  edge_6 jsonb,
  geometry geometry(Polygon, 4326) NOT NULL,
  properties_raw jsonb NOT NULL DEFAULT '{}'::jsonb,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

### 索引
```sql
CREATE INDEX hex_cells_dataset_idx ON hex_cells(dataset_id);
CREATE UNIQUE INDEX hex_cells_dataset_grid_uidx ON hex_cells(dataset_id, level, col, row);
CREATE INDEX hex_cells_geometry_gix ON hex_cells USING gist(geometry);
CREATE INDEX hex_cells_center_idx ON hex_cells(center_lon, center_lat);
```

说明：`properties_raw` 保留原始属性，避免后续发现 GeoJSON 还有重要字段时无法恢复。

## Edge 字段解析规则
1. 读取 polygon 外环坐标。
2. 去掉闭合重复点。
3. 校验有效六角格应有 6 个顶点；不是 6 个时记录 warning，并把 `edge_1..edge_6` 置空或尽量按已有边写入。
4. 计算 ring 方向，统一为逆时针顺序。
5. 生成 6 条边：每条边保存 `{ start: [lon, lat], end: [lon, lat], midpoint: [lon, lat] }`。
6. 找出 midpoint 纬度最大的边作为 `edge_1`。
7. 从 `edge_1` 开始按逆时针顺序填入 `edge_2..edge_6`。

## 后端 API 设计

### 导入 GeoJSON 到数据库
`POST /api/hex/import-file`

请求：
```json
{
  "filePath": "data/L4.geojson",
  "name": "taoyuan-l4"
}
```

响应：
```json
{
  "dataset": {
    "id": "uuid",
    "name": "taoyuan-l4",
    "featureCount": 500,
    "bounds": [minLon, minLat, maxLon, maxLat]
  }
}
```

### 列出数据集
`GET /api/hex/datasets`

### 读取数据集 GeoJSON
`GET /api/hex/datasets/:datasetId/geojson?minLon=&minLat=&maxLon=&maxLat=&limit=`

用于前端地图加载、zoom to layer 和导出。无 bbox 时返回整个数据集，但需要 limit 或后端上限保护。

### 删除数据集
`DELETE /api/hex/datasets/:datasetId`

默认先实现硬删除，因为用户需求是“可以删除”。如果后续要审计或防误删，可改为软删除 `deleted_at`。

### DEM 采样并回写数据库
`POST /api/hex/datasets/:datasetId/dem/sample-batch`

请求：
```json
{
  "batchIndex": 0,
  "datasourceId": "dem-srtm",
  "scale": 30
}
```

响应：
```json
{
  "features": [],
  "batchIndex": 0,
  "totalBatches": 10,
  "done": false
}
```

后端从数据库读取当前 batch 的 cells，调用 GEE 采样，更新 `hex_cells.dem`，并返回本批 GeoJSON features 给前端增量渲染。

### Landcover 采样并回写数据库
`POST /api/hex/datasets/:datasetId/landcover/sample-batch`

行为同 DEM，更新 `hex_cells.landcover`。

## 前端变更计划

### `front/src/api.js`
新增 API 封装：
- `importHexDataset({ filePath, name })`
- `fetchHexDatasets()`
- `fetchHexDatasetGeojson(datasetId, params)`
- `deleteHexDataset(datasetId)`
- `sampleDatasetDemBatch(datasetId, payload)`
- `sampleDatasetLandcoverBatch(datasetId, payload)`

### `front/src/App.vue`
1. 六角格操作区新增数据集名输入框。
2. `handleFileSelected(filePath)` 改为调用 `importHexDataset()`。
3. 导入成功后使用返回的 `dataset.id` 创建图层元数据。
4. 再调用 `fetchHexDatasetGeojson(dataset.id)` 获取可渲染 GeoJSON。
5. `hexagonLayers` 不再长期保存完整 GeoJSON，只保存：
   - `id`
   - `datasetId`
   - `name`
   - `visible`
   - `featureCount`
   - `bounds`
6. DEM / landcover 采样改为传 `datasetId`，前端不再重复上传完整 GeoJSON。
7. 导出时从数据库拉取完整 GeoJSON 或流式导出。
8. 删除图层时调用 `DELETE /api/hex/datasets/:datasetId`，成功后移除 OpenLayers 图层。

### `front/src/components/FilePickerDialog.vue`
保留文件上传和选择，不直接负责数据库导入。导入所需数据集名由 `App.vue` 管理。

### `front/src/components/LayerPanel.vue`
继续复用当前的导出、删除、zoom to layer 按钮。图层的 `canExport`、`canDelete`、`canZoom` 仍由 `App.vue` 计算。

### `front/src/useMap.js`
基本保留当前多图层结构。变化点：
- 添加数据库数据集 ID 与 OpenLayers layer ID 的映射只放在 `App.vue` 元数据中。
- `zoomToLayer()` 仍按已有 OpenLayers source extent 工作。
- 如果只加载 bbox 子集，zoom 全量范围应使用 `hex_datasets.bounds`，必要时新增 `zoomToExtent(bounds)`。

## 数据流优化

### 当前流程
`GeoJSON 文件 -> 后端读文件 -> 前端保存完整 GeoJSON -> 前端传完整 GeoJSON 给采样接口 -> 后端采样 -> 前端保存结果`

### 目标流程
`GeoJSON 文件 -> 后端入库 -> 前端只保存 datasetId -> 采样接口按 datasetId 从数据库取 batch -> 回写 dem/landcover -> 前端按 batch 渲染结果`

收益：
- 前端内存压力下降。
- 大 GeoJSON 不再反复在前后端传输。
- 刷新页面后数据集仍可恢复。
- 后续可以基于 PostGIS 做视图范围查询。

1. 六角格操作区增加数据集名输入。

## 风险与约束
1. 数据库 schema 变更属于项目红线，实施前需要明确确认。
2. 不修改真实 `.env`，只新增或更新 `.env.example` / 文档说明。
3. 如果本机没有 PostGIS，空间查询和 geometry 存储会阻塞；需要先安装扩展。
4. 大文件批量入库需要分批事务，不能一次性构造超大 SQL。
5. `edge_1..edge_6` 的“北方向开始、逆时针”依赖几何坐标顺序和六角格有效性，导入时必须记录异常 feature 数。
6. GEE 采样仍受认证、代理、配额、网络影响，数据库只解决持久化和传输效率。

## 不在本轮实现的内容
1. 多用户权限系统。
2. 异步任务队列。
3. 历史版本管理。
4. 每个数据集独立 PostgreSQL database。
5. 自动安装 PostgreSQL / PostGIS 或修改系统配置。
